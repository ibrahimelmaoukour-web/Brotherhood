# 🔒 Security Policy\n\n## Reporting Security Vulnerabilities\n\nIf you discover a security vulnerability in Brotherhood, please report it responsibly.\n\n### ⚠️ Do NOT\n\n- Create a public GitHub issue detailing the vulnerability\n- Post in discussions or forums\n- Share vulnerability details publicly\n- Attempt to exploit the issue\n\n### ✅ Do\n\n1. **Document** the vulnerability details:\n   - What is the vulnerability?\n   - How can it be reproduced?\n   - What is the impact?\n   - Do you have a suggested fix?\n\n2. **Report** privately to maintainers:\n   - Email: [Create an issue asking how to report securely]\n   - Include: Full details, reproduction steps, impact assessment\n\n3. **Wait** for response:\n   - Maintainers will acknowledge receipt within 48 hours\n   - We'll work on a fix\n   - You'll be credited (unless you prefer anonymity)\n\n4. **Coordinate** the disclosure:\n   - We'll patch the vulnerability\n   - We'll release a security update\n   - Then the issue can be disclosed publicly\n\n## Security Considerations\n\n### LocalStorage\n\nBrotherhood stores data in browser LocalStorage:\n- ✅ Encrypted in HTTPS\n- ⚠️ Accessible to any JavaScript on the page\n- ⚠️ Not encrypted if accessed over HTTP\n- ⚠️ Cleared if browser cache is cleared\n\n**Implication**: Do not store sensitive personal information.\n\n### Authentication\n\nThe authentication system is **simple and not cryptographic**:\n- It uses a hardcoded passphrase (\"FORGE\")\n- It's designed for community access control, not security\n- Don't use this for protecting sensitive data\n- Anyone with access to the code knows the password\n\n### Google OAuth\n\nThe OAuth button is **simulated**:\n- It does not actually connect to Google\n- No real authentication happens\n- It's a placeholder for future implementation\n\n### Best Practices for Users\n\nIf you host Brotherhood:\n1. Use HTTPS (encrypts LocalStorage)\n2. Don't add real authentication (it's not secure)\n3. Don't store sensitive personal data\n4. Keep dependencies updated\n5. Review code before deploying\n\n---\n\n## Vulnerability Response Timeline\n\n| Timeframe | Action |\n|-----------|--------|\n| **Day 0** | Report sent, maintainers acknowledge |\n| **Days 1-3** | Investigation & reproduction |\n| **Days 4-7** | Fix developed & tested |\n| **Day 8** | Patch released |\n| **Day 9+** | Public disclosure (optional) |\n\n## Credits\n\nResponsible security researchers will be:\n- ✅ Credited in security update\n- ✅ Listed in fame/honor section\n- ✅ Recognized in project documentation\n- ✅ Contacted for future collaboration\n\n---\n\n## Past Security Issues\n\n_None reported yet.
